# 작업 절차

v0.2 · 2026.07
규칙은 `publishing-rules.md` 참조. 이 문서는 **범위와 완료 기준**을 정한다.

---

## 1. 작업 단위

**1 작업 = 컴포넌트 1개** 또는 **규칙 1건**.
여러 컴포넌트를 한 번에 요청하지 않는다. 검수가 불가능해진다.

---

## 2. 요건 정의 (STEP 0)

작업 시작 전에 아래를 **먼저 채운다.** 비어 있는 항목이 있으면 시작하지 않는다.
여기서 안 적은 건 AI가 알아서 만들고, 그게 범위 이탈의 시작이다.

```
컴포넌트  :
변형      :   (예: primary / default / text / link)
상태      :   (예: default / hover / focus / disabled / loading)
사이즈    :   (예: sm / md / lg)
반응형    :   (대응 / 미대응, 대응이면 브레이크포인트)
접근성    :   (키보드 조작, aria 속성, focus 표시)
JS 동작   :   (있음 / 없음, 있으면 무엇)
참고      :   (기존 컴포넌트 중 재사용할 것)
```

> `table__sort`(정렬 가능한 헤더)는 STEP 0에서 누락돼 뒤늦게 추가됐다.
> 요건에서 빠진 항목은 나중에 반드시 다시 돌아온다.

---

## 3. 범위

### 포함

- 요건에 적힌 변형 · 상태 · 사이즈
- 해당 컴포넌트의 `css/component/*.css` + `component/*.html`
- 필요한 JS는 `js/main.js`에 추가

### 미포함 — 요청해도 이번 작업에서 하지 않는다

- 다른 컴포넌트 파일 수정
- `tokens.css` 값 변경 · 삭제
- 기존 코드 리팩터링
- 요건에 없는 변형 · 상태 추가
- 작업 중 발견한 다른 파일의 버그 수정

**미포함 항목이 필요해지면 별도 작업으로 분리한다.**

---

## 4. 절차

| STEP | 내용 | 산출 |
|---|---|---|
| 0 | 요건 정의 | 위 양식 작성 |
| 1 | 마크업 | `component/*.html` |
| 2 | 기본 스타일 | `css/component/*.css` |
| 3 | 상태 · 변형 채우기 | 요건의 상태/변형 전부 |
| 4 | 반응형 | 미디어쿼리 |
| 5 | 접근성 | focus-visible, aria, 키보드 |
| 6 | JS 동작 | `js/main.js` |
| 7 | 검수 | 아래 QA 통과 |
| 8 | 기록 | `TODO.md`에 결정 · 발견 사항 |

**각 STEP이 끝나면 멈추고 확인한다.** 0→8을 한 번에 요청하지 않는다.

---

## 5. 완료 기준

아래를 **전부** 충족해야 완료다. 하나라도 미충족이면 미완료로 둔다.

- [ ] 요건에 적은 변형 · 상태 · 사이즈가 전부 구현됨
- [ ] 데모 페이지에서 모든 변형이 한 화면에 보임
- [ ] QA 체크리스트 통과
- [ ] `TODO.md`에 결정 사항 기록
- [ ] 브라우저에서 실제로 눈으로 확인함

마지막 항목이 빠지면 무효 CSS를 못 잡는다.

## 6. QA

규칙 근거는 `publishing-rules.md`. 이 목록이 검수의 유일한 기준이다.

### 토큰

- [ ] 하드코딩된 색상 · 간격 값 없음
- [ ] 컴포넌트 CSS가 primitive 토큰을 직접 참조하지 않음
- [ ] 새 토큰이 기존 값과 중복 아님
- [ ] 토큰 이름이 숫자 표기 (`xs` `sm` `md` `lg` 안 씀)
- [ ] 토큰 이름에 대문자 없음
- [ ] `var()` 이름이 실제 존재하는 토큰인지 — 오타는 조용히 무시됨

### 단위

- [ ] font-size가 rem
- [ ] gap · padding · border가 px
- [ ] 미디어쿼리가 min-width · em
- [ ] 미디어쿼리가 해당 규칙 바로 아래 위치

### 레이아웃

- [ ] 바깥 반복이 grid, 안쪽 정렬이 flex
- [ ] grid 컬럼 수를 클래스가 아닌 `--cols`로 넘김

### 네이밍

- [ ] BEM 표기, 요소 2단계 이하
- [ ] 상태 클래스가 `is-` (`is--` 아님)
- [ ] `wrap`은 부모, `box`는 자식
- [ ] 줄임말이 고정 목록과 일치 — 줄임 `btn` `wrap` `desc` `thumb` `nav`
- [ ] kebab-case

### 코드 스타일

- [ ] 스페이스 2칸, 선택자 앞 공백 없음
- [ ] 속성이 배치 → 크기 → 타이포 → 색상 순으로 묶임
- [ ] 인라인 주석이 값이 아닌 이유를 설명함

### 마크업

- [ ] 이유 없는 div 없음
- [ ] id 중복 없음 (SVG `clipPath` 포함)
- [ ] 폼 안 버튼에 `type="button"`
- [ ] label ↔ input 연결
- [ ] 입력 타입이 용도와 일치 (tel / email / password)

### 이미지

- [ ] 아이콘 · 로고가 SVG
- [ ] 사진이 PNG 2배수(`@2x`)이고 표시 크기를 명시함

### 접근성

- [ ] 모든 인터랙티브 요소에 focus-visible
- [ ] disabled에 `cursor:not-allowed`
- [ ] 아이콘만 있는 버튼에 `aria-label`
- [ ] 키보드만으로 조작 가능
- [ ] 터치 영역 44px 이상

## 7. AI 요청 양식

```
[작업] {컴포넌트명} STEP {번호}

[요건]
(STEP 0 양식 붙여넣기)

[범위]
수정 대상 파일: {파일 경로만}
그 외 파일은 읽기만 하고 수정하지 않는다.

[규칙]
publishing-rules.md 를 따른다.

[금지]
- 요건에 없는 변형/상태 추가
- 지정하지 않은 파일 수정
- 기존 코드 리팩터링
- tokens.css 값 변경 (필요하면 제안만)
- 발견한 버그의 임의 수정

[보고]
작업 중 발견한 문제는 고치지 말고 목록으로 보고한다.
수정 여부는 내가 판단한다.
```

**마지막 두 줄이 핵심이다.** 발견과 수정을 분리하지 않으면 범위가 계속 넓어진다.

---

## 8. 발견 사항 처리

작업 중 다른 문제가 나왔을 때.

```
발견 → 보고 → 판단 → 별도 작업으로 등록 → 처리
```

**같은 작업 안에서 처리하지 않는다.**
급하면 지금 작업을 끝내고 바로 다음 작업으로 잡는다.

기록은 `TODO.md`에 날짜와 함께 남긴다.
"왜 그렇게 정했는지"까지 적는다. 결정만 적으면 나중에 근거를 다시 찾게 된다.
